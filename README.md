# Local Discord Voice Assistant

Прототип локального голосового Discord-бота:

```text
Discord voice -> Opus/PCM -> faster-whisper -> LM Studio -> Piper/Silero TTS -> Discord voice
```

Бот получает отдельный аудиопоток активного пользователя, завершает реплику после паузы, распознаёт русский текст и отвечает в тот же voice-канал. По умолчанию диалог начинается со слова `Омни`; после окончания озвучки следующие 30 секунд можно говорить без повторного обращения. Генерация, TTS и длительность ответа не вычитаются из этого окна: Discord-бот запускает backend-таймер только когда playback queue становится пустой. Вне активного follow-up gateway сначала проверяет PCM отдельной моделью hotword `small`: фоновая речь без wake word отбрасывается локально и не попадает в основную Whisper queue.

Backend использует динамический двухэтапный pipeline. Whisper распознаёт аудио независимо от занятой LM Studio, а генерация и TTS выполняются последовательно. Очередь распознавания хранит ограниченное число последних реплик каждого пользователя и обслуживает пользователей round-robin. Прямые обращения `Омни, ...` имеют приоритет над follow-up генерациями и удаляют устаревшие фоновые ответы.

Состояние pipeline доступно в `/health`: `recognition_active`, `recognition_queued`, `recognition_coalesced`, `recognition_cancelled`, `generation_active`, `generation_direct_queued`, `generation_followup_queued` и `generation_coalesced`. Совместимые агрегаты остаются в `active` и `queued`.

Отдельный Whisper `small` проверяет незавершённую реплику во время записи и сообщает боту о `Омни`. После окончания реплики gateway обязательно повторяет hotword-проверку по всему аудио с ожиданием свободного worker slot, поэтому временный `busy` ранней проверки не теряет обращение. Только после положительной проверки полная модель распознаёт реплику целиком. `WAKE_WORD_ALIASES` покрывает типичные ошибки Whisper, включая `помни`, `о мне`, `омли`, `омне`, `амни`, `умни` и близкие формы. Чем шире список, тем выше вероятность ложной активации на обычной речи.

## Требования

- Bun 1.2 или новее.
- Python 3.11-3.13.
- Запущенный LM Studio Local Server.
- Исходящий TCP/UDP-доступ к Discord.
- Discord-приложение с bot user и правами `View Channels`, `Connect`, `Speak`, `Use Application Commands`.

FFmpeg для основного аудиотракта не нужен: gateway и voice-core обмениваются сырым PCM.

## Настройка Discord

1. Создайте приложение в [Discord Developer Portal](https://discord.com/developers/applications).
2. На странице `Bot` создайте bot user и получите token.
3. В `OAuth2 -> URL Generator` выберите scopes `bot` и `applications.commands`.
4. Выдайте permissions `View Channels`, `Connect`, `Speak` и добавьте бота на сервер.
5. Скопируйте bot token в `DISCORD_TOKEN`, а ID тестового сервера в `DISCORD_GUILD_ID`.

Если регистрация команд возвращает `Missing Access`, bot user не добавлен на сервер из `DISCORD_GUILD_ID` или указан ID другого сервера.

`Message Content Intent` этому прототипу не требуется.

## Установка

```bash
cp .env.example .env
bun install

python3 -m venv .venv
.venv/bin/pip install --upgrade pip
.venv/bin/pip install -r voice_core/requirements.txt
.venv/bin/python -m piper.download_voices --data-dir models ru_RU-ruslan-medium
```

Заполните `.env`. Идентификатор загруженной модели LM Studio можно проверить так:

```bash
curl http://127.0.0.1:1234/v1/models
```

Перед запуском:

```bash
bash scripts/check-env.sh
```

Посмотреть доступные Whisper-модели и фактический размер загрузки:

```bash
PYTHONPATH=voice_core .venv/bin/python -m voice_core.whisper_models list
```

Скачать модель заранее с progress bar:

```bash
PYTHONPATH=voice_core .venv/bin/python -m voice_core.whisper_models download small
```

При запуске voice-core модель автоматически скачивается с тем же progress bar и загружается в память. HTTP-сервис станет готов принимать запросы только после завершения подготовки Whisper.
Модели с суффиксом `.en` предназначены только для английской речи. Для русского разумная отправная точка — `small`; при ограниченных ресурсах используйте `base`.

## Запуск

Терминал 1:

```bash
./scripts/start-backend.sh
```

Для разработки можно запустить bot и backend одной Bun-командой с автоматическим перезапуском после изменения исходников:

```bash
bun run dev
```

`bun --watch` полностью перезапускает процесс Discord-бота при изменениях в `src`, чтобы старые listeners и voice connections не оставались в памяти. Uvicorn следит только за Python-кодом в `voice_core/voice_core`. Процессы также доступны отдельно через `bun run dev:bot` и `bun run dev:backend`; остановка общего dev-скрипта посылает сигнал обоим процессам, поэтому бот успевает выйти из voice-канала.

Терминал 2:

```bash
bun start
```

В Discord выполните `/join`, затем скажите: `Омни, привет`. Фраза `Омни, стоп` закрывает активное окно диалога локально и не отправляется в LM Studio. Команда `/image` принимает JPEG, PNG или WebP до 10 МБ: после сигнала следующая голосовая реплика автора становится запросом к изображению без повторного `Омни`. `/voice` переключает движок и голос озвучки, `/effect` — профиль обработки голоса, `/reset` очищает историю, `/leave` отключает бота, `/status` проверяет voice-core.

LM Studio может вызывать встроенные backend-инструменты для актуальной погоды, краткой справки и анекдотов. `get_current_weather` использует Open-Meteo, `lookup_topic` — Wikimedia с резервным кратким ответом DuckDuckGo, а `get_random_joke` — JokeAPI в безопасном режиме с переводом результата моделью. Если пользователь уточняет тему анекдота, модель передаёт её английский поисковый вариант в фильтр JokeAPI; без темы выбирается случайная шутка. API-ключи для этих источников не нужны. Примеры: `Омни, какая сейчас погода в Москве?`, `Омни, расскажи про Эйнштейна`, `Омни, расскажи анекдот про программистов`.

Инструмент `search_web` ищет актуальные сведения в DuckDuckGo и передаёт модели до восьми заголовков, ссылок и сниппетов. Он доступен администраторам Discord-сервера автоматически. Обычный пользователь не получает этот tool в запросе LM Studio и при явной просьбе о поиске получает локальный отказ. Администраторы управляют постоянным whitelist через команды:

```text
/web-access add user:@пользователь
/web-access remove user:@пользователь
/web-access list
```

Discord скрывает команду от участников без `Administrator`, а bot дополнительно проверяет permission при выполнении. Whitelist хранится по `guild_id + user_id` в `USER_MEMORY_DB_PATH` и сохраняется после перезапуска. Пример голосового запроса: `Омни, поищи в интернете свежие новости о Python`.

Для `/image` в LM Studio должна быть загружена модель с поддержкой vision. Изображение прикладывается только к первой голосовой реплике после команды и удаляется из контекста после ответа. Следующие follow-up запросы получают обычную текстовую историю без изображения. В SQLite и на диск base64 не записывается.

`faster-whisper` скачивается и загружается при старте backend. Piper-голос загружается один раз командой установки выше. CPU-сборка Torch устанавливается из официального PyTorch index, указанного в `requirements.txt`.

## Настройка производительности

По умолчанию Whisper и Piper работают на CPU, чтобы не конкурировать с LM Studio за VRAM. Для NVIDIA GPU можно попробовать:

```dotenv
WHISPER_DEVICE=cuda
WHISPER_COMPUTE_TYPE=float16
```

На Windows `voice_core/requirements.txt` дополнительно устанавливает CUDA 12 runtime-библиотеки NVIDIA внутрь `.venv`. При загрузке Whisper backend автоматически регистрирует все каталоги `.venv/Lib/site-packages/nvidia/*/bin` через `os.add_dll_directory`, поэтому ручное изменение `$env:Path` после каждого перезапуска не требуется. Запускайте backend Python-интерпретатором именно из окружения:

```powershell
.\.venv\Scripts\python.exe -m uvicorn voice_core.app:app --app-dir voice_core --host 0.0.0.0 --port 8765
```

В логе успешного запуска появится `Registered Windows CUDA DLL directories`. Если каталогов нет, backend выведет отдельное предупреждение до загрузки Whisper.

Если распознавание на CPU медленное, используйте `WHISPER_MODEL=base`. Если качество русского недостаточно и хватает ресурсов, используйте `medium`.

Раннее распознавание wake word использует отдельную лёгкую модель и не ждёт завершения реплики:

```dotenv
HOTWORD_MODEL=small
HOTWORD_DEVICE=cpu
HOTWORD_COMPUTE_TYPE=int8
HOTWORD_CHECK_INTERVAL_MS=650
HOTWORD_MIN_AUDIO_MS=600
HOTWORD_TIMEOUT_MS=5000
```

Интервал задаёт частоту проверки текущего PCM-фрагмента. Уменьшение ускоряет реакцию, но увеличивает загрузку CPU. После раннего обнаружения бот сразу проигрывает сигнал, прерывает текущую озвучку и отменяет активную генерацию в voice-core; полная модель остаётся источником итогового текста.

По умолчанию `HOTWORD_MODEL=small`: это компромисс между задержкой и качеством распознавания короткого русского имени. Tiny и turbo для hotword не рекомендуются из-за конфликтов с похожими словами и артефактов. Модель использует два worker slot: ранние проверки не ждут занятый слот, а обязательные финальные проверки получают приоритет и дожидаются результата.

Чтобы Whisper не терял короткое wake word в начале фразы, перед аудио добавляется небольшой запас тишины. `hotwords`, `initial_prompt` и встроенный `vad_filter` фиксированно отключены: они конфликтуют с отдельным hotword-gate и Silero speech gate.

```dotenv
WHISPER_BEAM_SIZE=2
WHISPER_LEADING_SILENCE_MS=400
```

Пустые включения микрофона отбрасываются до любого partial hotword HTTP-запроса по уровню PCM, затем дополнительно проверяются быстрым Silero VAD до запуска основной модели Whisper:

```dotenv
DISCORD_MIN_VOICE_RMS=120
DISCORD_MIN_VOICE_PEAK=600
WHISPER_SPEECH_GATE=true
WHISPER_SPEECH_GATE_THRESHOLD=0.5
WHISPER_MIN_SPEECH_MS=180
```

Gateway допускает запись только после превышения обоих порогов: среднего уровня `DISCORD_MIN_VOICE_RMS` и пикового `DISCORD_MIN_VOICE_PEAK`. Пока пороги не достигнуты, аудио не отправляется даже в hotword Whisper. В `voice.capture.discarded_low_energy` выводятся реальные и минимальные значения, поэтому пороги можно подстроить под микрофон.

## Модели озвучки

Команда `/voice` показывает все Piper ONNX-файлы из каталога рядом с `PIPER_MODEL_PATH` и голоса Silero `v5_5_ru`: `aidar`, `baya`, `kseniya`, `xenia`, `eugene`. Выбор хранится в памяти отдельно для каждого Discord-сервера и после перезапуска backend сбрасывается на голос из `TTS_DEFAULT_VOICE`. Если переменная пуста, используется модель из `PIPER_MODEL_PATH`.

Голос по умолчанию задаётся полным идентификатором из списка `/voice`:

```dotenv
TTS_DEFAULT_VOICE=silero:xenia
```

Для Piper формат такой: `TTS_DEFAULT_VOICE=piper:ru_RU-ruslan-medium`. Неизвестный идентификатор останавливает запуск backend с перечнем доступных голосов.

При первом выборе Silero backend скачивает `models/v5_5_ru.pt` размером около 139 МБ с прогрессом в логах. Следующие переключения используют локальный файл и уже загруженную модель. Silero работает на CPU и синтезирует сразу в 48 кГц. Учитывайте лицензию модели при использовании вне личного прототипа.

Темп Piper и пауза между предложениями настраиваются в `.env`:

```dotenv
PIPER_LENGTH_SCALE=1.18
PIPER_SENTENCE_SILENCE_MS=320
```

Чем больше `PIPER_LENGTH_SCALE`, тем медленнее речь. Эти параметры не изменяют Silero. Для Silero включены автоматические ударения, восстановление `ё` и синтез отдельными законченными предложениями с короткой паузой:

```dotenv
SILERO_PUT_ACCENT=true
SILERO_PUT_YO=true
SILERO_SENTENCE_SILENCE_MS=220
```

Увеличивайте `SILERO_SENTENCE_SILENCE_MS`, если ответы всё ещё звучат торопливо. Пунктуацию внутри предложения, включая запятые и вопросительный знак, обрабатывает сама модель. Поле `reasoning_effort` в LM Studio по умолчанию не отправляется: обычные instruct-модели вроде Llama 3.2 его не поддерживают и иначе пишут WARN. Для reasoning-моделей задайте `LMSTUDIO_REASONING_EFFORT=none`, иначе весь лимит токенов может уйти на размышление и ответ останется пустым.

Русская модель Silero не произносит латиницу напрямую, поэтому перед синтезом английские фрагменты автоматически переводятся в произносимую кириллицу. Аббревиатуры вроде `API`, `CPU` и `GPT` читаются по английским названиям букв, распространённые технические слова используют заданную фонетику, остальные слова транслитерируются. Исходный текст ответа и сообщения в Discord-чате при этом не изменяется.

Команда `/effect` переключает постобработку отдельно для каждого Discord-сервера. Профиль `Роботизированный` добавляет понижение тона, тихую верхнюю гармонию, лёгкую амплитудную модуляцию, chorus, компрессию и короткий металлический reverb. Он работает поверх любого голоса, но наиболее выраженный результат получается с `silero:xenia` или `silero:kseniya`. По умолчанию обработка выключена:

```dotenv
TTS_DEFAULT_EFFECT=none
TTS_ROBOT_PITCH_SEMITONES=-1.5
TTS_ROBOT_HARMONY_VOLUME=0.10
TTS_ROBOT_MODULATION_HZ=35
TTS_ROBOT_MODULATION_DEPTH=0.07
TTS_ROBOT_REVERB=0.06
```

Для более механического звучания постепенно увеличивайте `TTS_ROBOT_MODULATION_DEPTH` и `TTS_ROBOT_HARMONY_VOLUME`. Большие значения быстро ухудшают разборчивость, поэтому backend ограничивает допустимые диапазоны и пропускает итог через limiter.

Перед Silero числа и распространённые единицы автоматически переводятся в произносимый русский текст. Например, `30 мл`, `70 кг` и `около 2 литров` превращаются в `тридцать миллилитров`, `семьдесят килограммов` и `около двух литров`. Нормализация выполняется только для TTS и не изменяет историю LM Studio.

Ответ LM Studio читается потоково. Пока модель работает, backend периодически выводит прошедшее время, приблизительное количество token chunks, долю лимита `LMSTUDIO_MAX_TOKENS`, скорость и размер уже полученного текста:

```dotenv
LMSTUDIO_PROGRESS_INTERVAL_SECONDS=2
```

Процент приблизительный: API не сообщает заранее длину будущего ответа, поэтому это доля уже полученных чанков от максимального лимита, а не точная оценка оставшегося времени.

Стоп-фразы задаются через запятую и обрабатываются до LM Studio. Во время follow-up достаточно сказать одну из коротких фраз; варианты со всеми wake aliases, например `О, мне, фатит` или `Помни, отключайся`, строятся автоматически:

```dotenv
STOP_PHRASES=стоп,стопэ,стопе,хватит,хвати,фатит,достаточно,харе,харэ,заканчивай,заканчиваем,закончи,закончить,закончим,закончили,заверши,завершить,заверши диалог,закрой диалог,прекрати,прекращай,прекрати слушать,перестань слушать,не слушай,замолчи,молчи,можешь отключаться,отключайся,остановись,останови диалог,до свидания,пока,на этом закончим,на этом всё,на этом все,давай закончим,давай завершим,закончим разговор,завершим разговор,закончим диалог,больше не слушай,хватит слушать,можешь отключиться
```

Локальная проверка игнорирует в stop-команде слова вежливости и заполнители `ну`, `пожалуйста`, `плиз`, `прошу`, `давай`, `ладно`, `всё`, поэтому `Омни, ну пожалуйста, хватит` совпадает с `омни хватит`, а `давай на этом закончим` — с `на этом закончим`. Если свободная формулировка всё же дошла до LM Studio, backend принудительно выбирает терминальный инструмент `end_conversation`. Модели запрещено требовать одно специальное слово. Tool закрывает follow-up, очищает ожидающее распознавание и generation-запросы, озвучивает короткое прощание (своё или из аргумента `farewell`) и отправляет боту `followup_stopped` с нисходящим сигналом.

При вызове других инструментов backend озвучивает нейтральный статус («Секунду, смотрю погоду», «Ищу в сети» и т.п.), либо короткую фразу модели, если она уже сказана перед tool call. Event `status_speech` приходит по WebSocket вместе с готовым PCM и проигрывается до финального ответа.

После обращения `Омни` follow-up окно открывается только для обратившегося пользователя. Вне активного окна, image-dictation и воспроизведения бот показывает себя self-muted в Discord. При активации follow-up микрофон размьючивается и остаётся в этом состоянии без прерываний до stop/таймаута; сигналы и ответы также временно размьючивают его. Получение входящего аудио при этом остаётся включённым через `selfDeaf: false`. Реплика считается попавшей в окно по моменту начала записи, поэтому медленное распознавание после дедлайна её не потеряет. Если бот закончил отвечать уже после истечения окна, follow-up запускается заново с полной длительностью. После стоп-фразы ожидающие распознавания этого пользователя удаляются. Короткие фоновые подтверждения отбрасываются до LM Studio:

Одиночное `Омни` без продолжения только активирует follow-up окно и сигнал готовности. До первой содержательной реплики это окно закреплено за позвавшим пользователем: gateway не начинает capture остальных участников, уже начатое параллельное аудио отбрасывается перед `/v1/turn`, а backend повторно проверяет ownership до generation queue. Блокировка снимается после содержательной follow-up реплики, stop или истечения окна. Повторное `Омни` того же активного пользователя не создаёт новый direct wake: слово вырезается, а остаток обрабатывается как followup. Generation job для LM Studio создаётся лишь после содержательной части обращения или следующей реплики владельца.

```dotenv
FOLLOWUP_SECONDS=30
FOLLOWUP_MIN_CHARS=4
FOLLOWUP_IGNORE_PHRASES=да,ага,угу,нет,ок,окей,ладно,понятно
```

## Звуковые сигналы

Backend отправляет события боту через WebSocket `/v1/events`. В voice-канале используются тихие сигналы и короткие статусные реплики:

- восходящий двойной тон — распознано прямое обращение `Омни, ...`;
- средний восходящий тон — изображение принято и можно надиктовывать запрос;
- протяжный высокий тон — распознана follow-up реплика;
- протяжный низкий тон — запрос дошёл до начала LM Studio/TTS генерации;
- нисходящий двойной тон — follow-up закрыт стоп-фразой или `end_conversation`;
- нисходящий тройной тон — время follow-up закончилось автоматически;
- `status_speech` — нейтральная озвучка действия инструмента или прощание при завершении диалога.

Громкость ограничена диапазоном от `0` до `0.2`, значение по умолчанию соответствует примерно 4.5% полной амплитуды:

```dotenv
AUDIO_CUE_VOLUME=0.045
```

## Память пользователей

Фразы вроде `Омни, называй меня Пупсик` обрабатываются через tool calling LM Studio. Модель вызывает backend-инструмент `remember_preferred_name`, а обращение сохраняется отдельно для сочетания Discord-сервера и пользователя в SQLite:

```dotenv
USER_MEMORY_DB_PATH=data/user-memory.sqlite3
```

При следующих обращениях модель получает сохранённое имя вместо технического Discord ID. Фраза `забудь, как меня называть` вызывает `forget_preferred_name`. Вопросы `как меня зовут` и `как зовут <участника>` принудительно вызывают `lookup_user_name`: инструмент требует явно указать `current_user` либо имя другого участника. При поиске другого человека backend исключает текущего говорящего из кандидатов, сам выбирает максимально похожую запись среди сохранённых обращений и последних Discord display name и требует ответить о найденном человеке в третьем лице. Сравнение учитывает ошибки и транслитерацию ника, например `формали бэд` сопоставляется с `formallybad`. Display name участников синхронизируются при `/join` и при входе новых пользователей в voice-канал, поэтому существующие preferred name связываются с никами по Discord user ID. Команда `/reset` очищает историю диалога, но не удаляет постоянную память пользователей.

Просьбы `отправь весь ответ в чат`, `продублируй ингредиенты в чат` и `отправь предыдущий ответ в чат` вызывают `send_message_to_chat`. Инструмент умеет отправить весь текущий ответ после завершения генерации, выбранную моделью часть или последний предыдущий ответ из истории. Сообщение публикуется в текстовый чат текущего voice-канала, mentions отключены, а текст длиннее лимита Discord делится по границам слов.

Внутри общей истории сервера каждый Discord user получает стабильный анонимный ключ `speaker_N`. Перед каждой генерацией LM Studio получает roster участников с их `identity_key`, текущим display name и сохранённым preferred name, а также отдельный `current_identity`. Поэтому одинаковые ники и параллельные обращения не смешиваются: история связывается с человеком по identity key, а не по отображаемому имени. Эти служебные ключи запрещено произносить system prompt, и перед TTS они дополнительно удаляются санитайзером. `/reset` создаёт новый реестр вместе с новой историей.

## Ограничения прототипа

- аудио нескольких говорящих захватывается независимо, а распознавание и генерация выполняются последовательно через backend-очередь;
- реплики длиннее `MAX_UTTERANCE_SECONDS` обрезаются до лимита, а не отклоняются;
- новое прямое обращение `Омни, ...` прерывает текущую генерацию и озвучку;
- история хранится в памяти и исчезает после перезапуска;
- сохранённые пользовательские обращения остаются в SQLite после перезапуска;
- ответ синтезируется полностью, потоковая озвучка предложений ещё не реализована;
- получение Discord audio зависит от актуальной поддержки DAVE в `@discordjs/voice`.

Участники voice-канала должны знать, что бот передаёт их речь в локальное распознавание. Аудио прототипом не сохраняется.
